ALib Compilation Symbols

Detailed Description

Symbols (C++ preprocessor macros) listed here, support adding or removing features to ALib when compiling the library and any code unit that uses ALib. These symbols must be used only within makefiles or IDE project files to be passed to the C++ compiler (!) (normally -D option). In other words: they must not be defined or undefined within the source code that uses ALib.

For most of the symbols, two versions with suffixes _ON and _OFF exists. These symbols decide about value of a corresponding code selection symbol that is named the same without the suffix.
If both are set at the same time, a compiler error is generated by ALib header files. If none of them is passed to the compiler, a default value is assumed.

As a convenience, to get default behavior, no symbol needs to be passed to the compiler.

Attention
Some of the compilation symbols are changing the definition of classes and methods which makes binaries incompatible with binaries that do not use the same setting for each of these symbols.
Symbols that have to be kept equal across compilation units are denoted in this documentation.

◆ ALIB_AVOID_ANALYZER_WARNINGS_ON

#define ALIB_AVOID_ANALYZER_WARNINGS_ON

This compiler symbol enables additional debug code that tries to prevent external analysis tools (e.g. Valgrind) to raise unnecessary warnings or errors.

When provided, it defines ALIB_AVOID_ANALYZER_WARNINGS.

◆ ALIB_DEBUG_OFF

#define ALIB_DEBUG_OFF

This compiler symbol forces ALib to compile without internal plausibility and assertion checks, by leaving the code selection symbol ALIB_DEBUG undefined. If neither this symbol nor ALIB_DEBUG_OFF is provided, then ALib headers rely on a set of standard symbols introduced (and set) by other libraries, e.g. NDEBUG.

◆ ALIB_DEBUG_ON

#define ALIB_DEBUG_ON

This compiler symbol forces ALib to compile in debug mode by defining the code selection symbol ALIB_DEBUG. Plausibility and assertion checks are performed in this mode.
If neither this symbol nor ALIB_DEBUG_OFF is provided, then ALib headers rely on the symbol NDEBUG. If NDEBUG is given, ALIB_DEBUG is not set. Otherwise it is.

#define ALIB_DEBUG_STRINGS_OFF

Disables certain debug code within class AString. See ALIB_DEBUG_STRINGS_ON. This symbol represents the default behavior and is provided for completeness.

◆ ALIB_DEBUG_STRINGS_ON

#define ALIB_DEBUG_STRINGS_ON

This compiler symbol enables additional debug code within class AString. When provided, it defines ALIB_DEBUG_STRINGS to true. This is useful when extending or specifically debugging class AString.

◆ ALIB_FEAT_BOOST_REGEX_ON

#define ALIB_FEAT_BOOST_REGEX_ON

This compiler symbol enables the use of string utility class RegexMatcher by defining source selection symbol ALIB_FEAT_BOOST_REGEX. Class RegexMatcher is in fact only a simple wrapper around external boost::regex library.

If this symbol is not explicitly given to the compiler, ALIB_FEAT_BOOST_REGEX remains undefined, the boost library is not included and class RegexMatcher is not available.

◆ ALIB_FEAT_EXPRESSIONS_SPIRIT_PARSER_ON

#define ALIB_FEAT_EXPRESSIONS_SPIRIT_PARSER_ON

This compiler symbol enables the use of alternative an an alternative parser for module ALib Expressions based on boost::spirit.

Note that setting this flag does not only choose to compile the boost::spirit based parser, but also choose to use it by default.

Attention
This is an experimental feature only. Some features of this library will be silently (!) lost with its activation. It is discouraged to use the boost::spirit parser in combination with this library in production environments!
Please read the according manual section for more information on the use of the boost::spirit parser.

◆ ALIB_FEAT_THREADS_OFF

#define ALIB_FEAT_THREADS_OFF

This compiler symbol disables the use of threads within ALib, by preventing the definition of source selection symbol ALIB_FEAT_THREADS. This might be useful for platforms where no thread library is available or to minimize code size. In general, if unsure, do not use this symbol and use the multi-threaded version of ALib.
When this symbol is set, classes Thread and Runnable are not available. However, the classes ThreadLockNR and ThreadLock are! Their interface remains the same, but they do not perform any locking. Due to this behavior, it is possible to write code that performs due locking while still, in single threaded compilation, the code does not suffer any performance drawbacks, because lock code will be pruned.

◆ ALIB_FEAT_THREADS_ON

#define ALIB_FEAT_THREADS_ON

This compiler symbol enables the use of threads within ALib, by. defining source selection symbol ALIB_FEAT_THREADS. This is the default, hence the symbol is not needed to be passed, but it is available for completeness.
See ALIB_FEAT_THREADS_OFF for more information.

◆ ALIB_NARROW_STRINGS_OFF

#define ALIB_NARROW_STRINGS_OFF

This compiler symbol forces the use of wide strings (character type wchar_t) as the default string type of the ALib library.
While still both string types are available, all neutral string alias types will use wide characters.

See also
  • ALIB_NARROW_STRINGS_ON
  • ALIB_NARROW_STRINGS
  • Manual section ALib String Character Types.

◆ ALIB_NARROW_STRINGS_ON

#define ALIB_NARROW_STRINGS_ON

This compiler symbol forces the use of narrow strings (character type char) as the default string type of the ALib library.
While still both string types are available, all neutral string alias types will use single byte characters.

If neither this macro nor ALIB_NARROW_STRINGS_OFF is given, the choice of default character type is system dependent: On Windows OS, wide characters strings are chosen, while on Unix-like OSes, narrow character strings are used by default.

See also
  • ALIB_NARROW_STRINGS_OFF
  • ALIB_NARROW_STRINGS
  • Manual section ALib String Character Types.

◆ ALOX_DBG_LOG_CI_OFF

#define ALOX_DBG_LOG_CI_OFF

Disables the generation of scope information using the built-in preprocessor macros like "__FILE__" and "__LINE__" for ALox debug logging statements

Passing this Symbol to the compiler, makes ALox to leave the code selection symbol ALOX_DBG_LOG_CI undefined.
Must not be used in conjunction with ALOX_DBG_LOG_CI_ON.

If neither ALOX_DBG_LOG_CI_ON nor ALOX_DBG_LOG_CIO_OFF are specified, then ALOX_DBG_LOG_CI_ON is the default.

Note
The ALox Scope Domain mechanism as well as the ALox features Lox.Once and Log Data rely on caller information. These mechanisms are therefore not available to debug logging when this symbol is set.

◆ ALOX_DBG_LOG_CI_ON

#define ALOX_DBG_LOG_CI_ON

Explicitly enables the generation of scope information using the built-in preprocessor macros like "__FILE__" and "__LINE__" for ALox debug logging statements

Passing this Symbol to the compiler, makes ALox define the code selection symbol ALOX_DBG_LOG_CI. Must not be used in conjunction with ALOX_DBG_LOG_CI_OFF.

If neither ALOX_DBG_LOG_CI_ON nor ALOX_DBG_LOG_CIO_OFF are specified, ALox will enable caller source information for debug logging. In other words, ALOX_DBG_LOG_CI_ON is the default and passing the symbol to the compiler is not mandatory.

Note
The ALox Scope Domain mechanism as well as the ALox features Lox.Once and Log Data rely on caller information. These mechanisms are therefore not available to debug logging unless this symbol is set.

◆ ALOX_DBG_LOG_OFF

#define ALOX_DBG_LOG_OFF

Explicitly disables debug logging by leaving the code selection symbol ALOX_DBG_LOG undefined. Any debug Log Statements that are embedded in code by using the ALox Macros For Debug Logging, will get pruned.

Must not be used in conjunction with ALOX_DBG_LOG_ON.

If neither ALOX_DBG_LOG_ON nor ALOX_DBG_LOG_OFF are specified, ALox will determine the inclusion of debug Log Statements by evaluating macro ALIB_DEBUG. If ALIB_DEBUG is set, ALOX_DBG_LOG will also be set.

◆ ALOX_DBG_LOG_ON

#define ALOX_DBG_LOG_ON

Explicitly enables debug logging by defining the code selection symbol ALOX_DBG_LOG.

Must not be used in conjunction with ALOX_DBG_LOG_OFF.

If neither ALOX_DBG_LOG_ON nor ALOX_DBG_LOG_OFF are specified, ALox will determine the inclusion of debug Log Statements by evaluating macro ALIB_DEBUG. If ALIB_DEBUG is set, ALOX_DBG_LOG will also be set.

◆ ALOX_REL_LOG_CI_OFF

#define ALOX_REL_LOG_CI_OFF

Disables the generation of scope information using the built-in preprocessor macros like "__FILE__" and "__LINE__" for ALox release logging statements

Passing this Symbol to the compiler, makes ALox to leave the code selection symbol ALOX_REL_LOG_CI undefined.
Must not be used in conjunction with ALOX_REL_LOG_CI_ON. If neither ALOX_REL_LOG_CI_ON nor ALOX_REL_LOG_CIO_OFF are specified, then ALOX_REL_LOG_CI_OFF is the default. In other words: passing the symbol to the compiler is not mandatory.

Note
The ALox Scope Domain mechanism as well as the ALox features Lox.Once and Log Data rely on caller information. These mechanisms are therefore not available to release logging unless ALOX_REL_LOG_CI_ON is set.

◆ ALOX_REL_LOG_CI_ON

#define ALOX_REL_LOG_CI_ON

Explicitly enables the generation of scope information using the built-in preprocessor macros like "__FILE__" and "__LINE__" for ALox release logging statements

Passing this Symbol to the compiler, makes ALox define the code selection symbol ALOX_REL_LOG_CI.
Must not be used in conjunction with ALOX_REL_LOG_CI_OFF.

If neither ALOX_REL_LOG_CI_ON nor ALOX_REL_LOG_CIO_OFF are specified, ALox will disable caller source information for release logging.

Enabling source info for for release logging is seldom wanted. Release executables should not generate log output that an end user is not able to understand. It can make sense however, if release log information from the field goes back to the software development team. Furthermore, it can be very helpful to enable scope information for release Log Statements in the debug version of a compilation unit.

Note
The ALox Scope Domain mechanism as well as the ALox features Lox.Once and Log Data rely on caller information. These mechanisms are therefore not available to debug logging unless this symbol is set.

◆ ALOX_REL_LOG_OFF

#define ALOX_REL_LOG_OFF

Explicitly disables release logging by leaving the code selection symbol ALOX_REL_LOG undefined. Any release Log Statements that are embedded in code by using the ALox Macros For Release Logging, will get pruned.

Must not be used in conjunction with ALOX_REL_LOG_ON.

If neither ALOX_REL_LOG_ON nor ALOX_REL_LOG_OFF are specified, release logging implemented by ALox release log macros is enabled. Therefore, this symbol is provided only for completeness. There is no real need to pass it to the compiler.

Remember, that any release logging that is not implemented using the ALox release logging macros, but instead using the ALox classes in the source code directly, is not affected by this macro.
