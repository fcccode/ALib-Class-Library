47 /**
48  * This namespace of \alib provides types for calendrical date and time processing as well as
49  * for non-calendrical steady and monotonic time measurement.
50  *
51  * Two central classes are provided to represent "point in times" or "time stamps":
52  * 1. Class \alib{time,DateTime} for calendrical values
53  * 2. Class \alib{time,Ticks} for steady and monotonic values
54  *
55  * Both classes allow to measure (probe) the current point in time, which is internally using a
56  * system-dependent clock source. With class \b %DateTime, the clock used is related to the system's
57  * clock setting, locale setting, daylight saving, etc., while with class \b %Ticks, the clock is
58  * steady and monotonic. Besides that, both classes are derived from template type
59  * \alib{time,TimePointBase} which provides basic functionality to work with clock values
60  * and "durations" (see inner type \alib{time::TimePointBase,Duration} of class
61  * \alib{time,TimePointBase}).
62  *
63  * Furthermore this namespace incorporates types used for conversion of "time points" and
64  * durations to values understandable to humans, hence calendar dates and clock values in hours,
65  * minutes, etc.<br>
66  *
67  * While for \alib{time::TimePointBase,Duration,durations} of both types (calendrical and
68  * steady/monotonic), class \alib{time,CalendarDuration} can be used to convert values to and from
69  * human understandable units, a similar class \alib{time,CalendarDateTime} allows only the
70  * conversion of \b %DateTime values to and from units understandable by humans. The reason for
71  * this is that values of type \b %Ticks can not be converted without ambiguities: Their conversion
72  * can only be performed using the current system clock, which means that changes of the current
73  * clock setting changes the result of the conversion!<br>
74  * As it still might be suitable to convert \b Ticks to human-readable calendar dates and time,
75  * class \alib{time,TicksConverter} is provided. This class allows to convert values of type
76  * \b %Ticks to values of type \b %DateTime, which then can be used with \b %CalendarDateTime.
77  * Of-course, it is up to the user of the library, to assure that ambiguities are solved,
78  * respectively it is duly made transparent to the end-user how such translated values are to
79  * be interpreted.
80  *
81  * \note
82  * A typical sample that makes the issues of conversion of point in time values more clear, is
83  * found in the area of activity logging of a software. Module \alibmod_alox provides tools to
84  * implement such application logging. As it is important to have a steady monotonic relationship
85  * between each log entry in respect to the moment of their occurrence, the library stores a
86  * \b %Ticks value with each entry. In many cases, end-users are only interested in two sorts of
87  * information:
88  * - The point in time given as a duration value relative to the start of the application
89  * - The point in time given as a duration value relative to the previous log statement.
90  *
91  * \note
92  * This means that the "absolute", calendrical point in time is not of interest in most cases.
93  * Nevertheless, the library allows to display also calendar dates and "time of day clock-times"
94  * if wished. Now, still such display should rather show "steady monotonic" times when listed.
95  * Therefore, no \b %DateTime values are stored with log entries, and if calendrical display
96  * is needed, a conversion using class \b %TicksConverter is performed. The object used for this
97  * is created only once and in favour to provide steady log dates, is not updated by the library.
98  * This means, if a software runs for a longer time, changes of the system clock are \b not
99  * reflected and therefore, the calendrical date and time values displayed may start to deviate
100  * from the true system clock!<br>
101  * A software that is using the library has to decide to either actively
102  * \alib{time,TicksConverter::SyncClocks,update the converter} (and hence introduce non steady
103  * log entries) or to acknowledge the user of the software of the problem of interpreting the
104  * date/time values!
105  */
