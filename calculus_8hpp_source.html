39 /** ************************************************************************************************
40  * ## 1. Introduction ##
41  * This struct specializes \b %CompilerPlugin and provides generic approaches and implementation
42  * of variants of method \b %TryCompilation.
43  * Consequently, this is the base type of most built-in compiler plug-ins, and is strongly
44  * recommended to consider this struct as the preferred base when implementing custom plug-ins.
45  *
46  * The class implements virtual, overloaded methods
47  * - \alib{expressions::CompilerPlugin,TryCompilation(CIFunction&)},
48  * - \alib{expressions::CompilerPlugin,TryCompilation(CIUnaryOp&)} and
49  * - \alib{expressions::CompilerPlugin,TryCompilation(CIBinaryOp&)}.
50  *
51  * For each <em>AST</em> node type to compile, this class holds a vector or a hash map to store
52  * all information needed for compilation.
53  *
54  * The simple schema of using this class is:
55  * 1. Derive a custom type from this class.
56  * 2. Create constant value objects and C++ native callback functions.
57  * 3. Create constant (\c constexpr) tables with all compilation information.
58  * 4. In the constructor of the derived class, feed the tables into to this struct, using the
59  * corresponding helper methods.
60  *
61  * \note
62  * Sample code and detailed explanations are given with user manual section
63  * \ref alib_expressions_cpcc_calculus "5.4 Class Calculus"
64  * and tutorial section
65  * \ref alib_expressions_tut_ffext "6. Extending The File Filter Sample"
66  *
67  * <p>
68  * \note
69  * The name "Calculus" seems a little overstated, we admit.
70  *
71  * ## 2. Choose Anonymous Namespaces ##
72  *
73  * A good design pattern is to put the callback functions, constant values and compilation information
74  * tables into an anonymous namespace at the start of the compilation unit (.cc or .cpp file) of
75  * your custom plug-in. This way, the linker is not bothered with the function and object names,
76  * which solely are referenced by their address and never need to get linked to other units.
77  *
78  * Just after this anonymous namespace, the implementation of the constructor of the custom
79  * plug-in, should be placed. As explained above, its duty is to fills in the vector/hash map using
80  * the right combination of operator or function argument types, as well as providing a proper
81  * \ref alib_expressions_prereq_sb "sample box" that corresponds to the type of the
82  * output parameter of the native function.
83  *
84  * When implementing a custom plug-in, it may be helpful to have a look at the source code of
85  * the built-in plug-ins provided with module \alibmod_expressions.
86  * You will see that these implementations are quite straight forward and use some 'handy' local
87  * preprocessor macros that may be copied and used for custom implementations.
88  *
89  * ## 3. Specifics For Identifiers and Functions ##
90  * While parent class \b %CompilerPlugin does not make any direct distinction between functions
91  * that are always returning a constant value and those that don't, this class does.
92  * Arguably such functions are always parameterless, hence identifiers. Samples for such constants
93  * are \b "PI" or \b "True".<br>
94  * Such constant identifiers are supported by populating table #ConstantIdentifiers which is a
95  * \c std::vector of elements of type \b %ConstantIdentifiersEntry.
96  * For details on each "table column", refer to the documentation of the fields of
97  * \alib{expressions::plugins::Calculus,ConstantIdentifierEntry}.
98  *
99  * Non-constant identifiers and functions are supported by populating table
100  * #Functions, which is a \c std::vector of elements of type \b %FunctionEntry.
101  * For details on each "table column", refer to the documentation of the fields of
102  * \alib{expressions::plugins::Calculus,FunctionEntry}.
103  *
104  * Some notes:
105  * - It can be specified if function names are case sensitive and whether they might be abbreviated.
106  * - The list of arguments (their types) accepted by a function is to be provided as a
107  * \c std::vector of \ref alib_expressions_prereq_sb "sample boxes". It is
108  * recommended to define such vectors once per unique function "signature" in the anonymous
109  * namespace section of the compilation unit and fill its contents once in the constructor of the
110  * custom plug-in. Such vectors can then be reused for each function that shares the same
111  * signature.
112  * - Variadic functions are supported by adding a final \e nulled \b %Box to the argument list.
113  * All sample argument types before this box are mandatory, but an arbitrary amount of arguments
114  * of likewise arbitrary type may be followed. It is also allowed to add just that one nulled
115  * \b %Box to the signature vector, which leads to functions that accept just any number of any
116  * type of argument, including zero arguments.
117  * - In debug compilations, besides the callback function pointer, the C++ name of the callback
118  * function is to be provided. For this, macro \ref CALCULUS_CALLBACK is defined.
119  * The macro creates a stringified version of the given function pointer, separated by a comma.
120  * - Flag \alib{expressions::plugins::Calculus,FunctionEntry::IsCTInvokable} is a boolean value
121  * that denotes whether a function can be evaluated at compile time in the case that all of the
122  * parameters given in the expression are constant.
123  * If so, this struct will do the invocation at compile time and return the constant result value
124  * instead of the function call.<br>
125  * Most built-in functions are compile time invokable. For example most mathematical functions
126  * like \c log(Float) or \c sin(Float) can be evaluated at compile time (again, only in the case
127  * that the given parameters are constant). The reason is, that
128  * these functions are independent from custom scope data.
129  * In contrast to this, custom functions, especially even parameterless identifiers usually are
130  * dependent on scope information and thus often can not be evaluated at compile time.
131  *
132  * ## 4. Specifics For Unary Operators ##
133  *
134  * The compilation of \alib{expressions,DefaultUnaryOperators,unary operators} is supported by populating nested
135  * hash map #UnaryOpMap. Because feeding a nested hash-map is a rather complicated process that
136  * can not easily be done using static table data, several helpers are provided:
137  * - #AddUnaryOp<br>
138  * This function adds compilation information for a single unary operator to #UnaryOpMap.
139  *
140  * - #UnaryOpTableEntry<br>
141  * This is a type definition that allows to define tables with compilation information on unary
142  * operators. It is recommended to create such tables as \c constexpr data in an anonymous
143  * namespace of the compilation unit.
144  *
145  * - #AddUnaryOps<br>
146  * This is a pair of overloaded functions. One of them is templated and just used to deduce
147  * the length of the given table of static data. This table is then fed as a pointer, together
148  * with the table size to the second method, which in turn feeds the table entries into
149  * #UnaryOpMap.
150  *
151  * In other words: Method #AddUnaryOp feeds one single operator to the nested hash-map, while
152  * #AddUnaryOps feeds static "bulk" data to it. For details of the functions and types, consult
153  * the corresponding documentation.
154  *
155  * As documented in user manual section
156  * \ref alib_expressions_operators_aliases "9.4 Type-Specific Operator Aliases",
157  * module \alibmod_nolink_expressions supports the use of alias
158  * operators. For unary operators, this is reflected by this class with:
159  *
160  * - #UnaryOpAliases<br>
161  * A nested hash map that collects information about unary operator aliases.
162  *
163  * - #AddUnaryOpAlias<br>
164  * This function adds information about an unary operator alias to #UnaryOpAliases.
165  *
166  * - #UnaryOpAliasTableEntry
167  * A type definition that allows to define tables with information about unary operator aliases.
168  * It is recommended to create such tables as \c constexpr data in an anonymous
169  * namespace of the compilation unit.
170  *
171  * - AddUnaryOpAliases<br>
172  * This is a pair of overloaded functions. One of them is templated and just used to deduce
173  * the length of the given table of static data. This table is then fed as a pointer, together
174  * with the table size to the second method, which in turn feeds the table entries into
175  * #UnaryOpAliases.
176  *
177  *
178  * ## 5. Specifics For Binary Operators ##
179  *
180  * The compilation of binary operators is supported by populating nested hash map #BinaryOpMap.
181  * Because feeding a nested hash-map is a rather complicated process that
182  * can not easily be done using static table data, several helper methods are provided:
183  * - #AddBinaryOp<br>
184  * This function adds compilation information for a single binary operator to #BinaryOpMap.
185  *
186  * - #BinaryOpTableEntry<br>
187  * This is a type definition that allows to define tables with compilation information on binary
188  * operators. It is recommended to create such tables as \c constexpr data in an anonymous
189  * namespace of the compilation unit.
190  *
191  * - #AddBinaryOps<br>
192  * This is a pair of overloaded functions. One of them is templated and just used to deduce
193  * the length of the given table of static data. This table is then fed as a pointer, together
194  * with the table size to the second method, which in turn feeds the table entries into
195  * #BinaryOpMap.
196  *
197  * In other words: Method #AddBinaryOp feeds one single operator to the nested hash-map, while
198  * #AddBinaryOps feeds static "bulk" data to it. For details of the functions and types, consult
199  * the corresponding documentation.
200  *
201  * As documented in user manual section
202  * \ref alib_expressions_operators_aliases "9.4 Type-Specific Operator Aliases",
203  * module \alibmod_nolink_expressions supports the use of alias
204  * operators. For binary operators, this is reflected by this class with:
205  *
206  * - #BinaryOpAliases<br>
207  * A nested hash map that collects information about binary operator aliases.
208  *
209  * - #AddBinaryOpAlias<br>
210  * This function adds information about a binary operator alias to #BinaryOpAliases.
211  *
212  * - #BinaryOpAliasTableEntry<br>
213  * A type definition that allows to define tables with information about binary operator aliases.
214  * It is recommended to create such tables as \c constexpr data in an anonymous
215  * namespace of the compilation unit.
216  *
217  * - #AddBinaryOpAliases<br>
218  * This is a pair of overloaded functions. One of them is templated and just used to deduce
219  * the length of the given table of static data. This table is then fed as a pointer, together
220  * with the table size to the second method, which in turn feeds the table entries into
221  * #BinaryOpAliases.
222  *
223  * \note
224  * It is \e not necessary to define an alias operator <c>'='</c> for operator <c>'=='</c>, when
225  * using this class, because this alias replacement is internally always made by this class, if
226  * compilation flag \alib{expressions,Compilation::AliasEqualsOperatorWithAssignOperator} is set
227  * in field \alib{expressions,Compiler::CfgCompilation}.
228  *
229  * \note
230  * In contrast to this, compilation flag \alib{expressions,Compilation::AllowBitwiseBooleanOperations}
231  * affects only built-in type \e boolean - just as the flag's name suggests.
232  * The flag is tested only in derived plug-in \alib{expressions,plugins::Arithmetics}.
233  * In other words: to allow for example operator <c>'&'</c> to be used as an alias for operator
234  * <c>'&&'</c> defined on custom types, this has to be explicitly added as an alias definition.
235  *
236  * Finally, for binary operators, this type provides a mechanism to provide information on
237  * compile-time optimizations. The following fields and methods are provided:
238  *
239  * - #BinaryOpConstLHSOptimizations<br>
240  * A nested hash-map that collects information about possible optimizations of binary operations
241  * when the left-hand side operator is a specific constant value.
242  *
243  * - #BinaryOpConstRHSOptimizations<br>
244  * A nested hash-map that collects information about possible optimizations of binary operations
245  * when the right-hand side operator is a specific constant value.
246  *
247  * - #BinaryOpOptimizationsTableEntry<br>
248  * A type definition that allows to define tables with information about binary operator
249  * optimizations..
250  * It is recommended to create such tables as \c constexpr data in an anonymous
251  * namespace of the compilation unit.
252  *
253  * - #AddBinaryOpOptimizations<br>
254  * A pair of overloaded functions. One of them is templated and just used to deduce
255  * the length of the given table of static data. This table is then fed as a pointer, together
256  * with the table size to the second method, which in turn feeds the table entries into
257  * either #BinaryOpConstLHSOptimizations or #BinaryOpConstRHSOptimizations.
258  *
259  * Samples of binary operator optimizations are given in documentation of struct
260  * \alib{expressions,CompilerPlugin::CIBinaryOp}, which is filled with the optimization
261  * information given by the use of above listed helpers.
262  *
263  * # Reference Documentation #
264  **************************************************************************************************/
