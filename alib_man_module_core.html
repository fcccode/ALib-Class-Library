Module "Core"

Not A "Real" Module

Modularization of a class library is not always 100% possible (or the effort is sometimes not worth the cost). Therefore, some of the basic, core classes can not be well assigned to a single ALib Module. They are included in the distribution as soon as other modules that need these classes are selected. The other way round, it is furthermore not documented too well, which of the low level ALib modules, and which of each other, these "core classes" depend upon.

Now, to be able to select all of these classes (to become included in an ALib distribution, respectively to become known to the build system), module ALib Core was defined. In other words: This ALib module does not provide specific extra functionality, but virtually gives a home to otherwise unassigned ALib classes.

Namespace List

When ALib Core is selected, all types of the following namespaces are included in the build:

  • aworx::lib::debug
  • aworx::lib::lang
  • aworx::lib::util
  • aworx::lib::threads
  • aworx::lib::time
  • aworx::lib::system

In addition, ALib Core selects modules ALib Singleton, ALib Boxing and ALib Strings, which some of the classes depend upon.

Note
The only module, apart from the three named above, that does not depend on this module is ALib Configuration, which selects only certain types from the namespaces listed above. All "higher" modules (ALox, ALib Expressions, ALib CLI, etc.) will select this module to be included.

Selected Types

The complete reference documentation of all the types are found with the list of namespaces given above. Nevertheless, some of the types should be highlighted here:

  • Class Exception, which implements a combination of two different paradigms of exception handl√≠ng.
  • Various tools to extend the features of C++ 11 enumerations (which are quite limited otherwise). This includes some template meta programming and other tricks, implemented with types Enum, EnumMetaData, T_EnumIsArithmetical, T_EnumIsBitwise, T_EnumIsIteratable and related types and functions.
  • Class Resources, which externalizes strings (while being able to adopt to a different, 3rd party system).
  • Class Library, which allows to share resources between libraries and organizes their bootstrap process.
  • Classes ThreadLockNR, ThreadLock and SmartLock and corresponding tool class Owner which are simple wrappers for functionality around std::mutex.
  • Class Ticks and sibling types of namespace time, which wrap and simplify functionality of namespace std::chrono.
  • Class MemoryBlocks, which allows to perform efficient memory allocations "en bloc".
  • Class StringTree, which is a generic container class, dedicated to store data along "string paths", similar to file systems addressing schemes.
