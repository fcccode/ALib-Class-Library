201 /** ************************************************************************************************
202  * Writes formatted text into an \alib{strings,AStringBase,AString}.
203  *
204  * \note
205  * The C++ version of this class relies heavily on classes found in module (namespace)
206  * \ref aworx::lib::boxing "ALib Boxing". In short, <b>%ALib %Boxing</b> allows to create methods
207  * that accept any type of argument as parameter, without the need of type conversion,
208  * wrapper classes, etc. Furthermore <b>%ALib %Boxing</b> allows to add virtual interfaces
209  * to C++ types, independent from the original implementation of the type itself. This means
210  * that any 3rd-party type can be adopted to support "boxing" and hence to be formatted with this
211  * class - without changing their implementation and with no access to their source code!<br>
212  * Knowledge about <b>%ALib %Boxing</b> is not needed to use this class with standard C++ types.
213  * Only if a string representation of custom types should be generated by this class with
214  * passing such custom types "as is", then support for <b>%ALib %Boxing</b> in respect to such
215  * custom types have to be added.
216  *
217  * \note
218  * The \ref aworx::lib::boxing "detailed documentation" of <b>%ALib %Boxing</b> describes how
219  * to do this. The "boxing interfaces" that are needed to be implemented for custom types
220  * to work with this formatters are
221  * \ref aworx::lib::strings::boxing::IApply "IApply" and optionally
222  * \ref aworx::lib::strings::boxing::IFormat "IFormat". The latter is used with
223  * \alib{strings::format,FormatterPythonStyleBase,FormatterPythonStyle}, if a custom format
224  * specification is given for a replacement field.
225  *
226  * ##Different Supported Formatting Syntax##
227  *
228  * While this class is abstract, only derived classes that support a specific format syntax can
229  * be instantiated and used. Built-in formatters provided with \alib are:
230  * - \alib{strings::format,FormatterPythonStyleBase,FormatterPythonStyle}, which mimics
231  * the [Python formatting syntax](https://docs.python.org/3.5/library/string.html#format-string-syntax)
232  * and
233  * - \alib{strings::format,FormatterJavaStyleBase,FormatterJavaStyle}, which mimics
234  * the formatting syntax of corresponding
235  * [Java class "Formatter"](http://docs.oracle.com/javase/8/docs/api/java/util/Formatter.html)
236  *
237  * One of the design goals of \alib is to provide a library for (currently) three programming
238  * languages \b C++, \b Java and \b C#, which allows to reach a certain level of source code
239  * compatibility if used in cross-platform/language projects. Now, why do we provide
240  * <b>Python Style</b> instead of to mimic what is coming with \b .Net languages like \b C#?
241  *
242  * The answer is that we considered <b>Python Style</b> to be more powerful. It is also more powerful
243  * than the <b>Java Style</b> formatting syntax. But the latter is in its basics similar to
244  * good old \c printf format string style and hence for \alib a must have.
245  * The good news is that in its very basics, <b>Python Style</b> is similar to \b .Net formatting.
246  * This way, there is some "familiar basic syntax" available for everybody.<br>
247  * Finally, it is worth to mention that the \alib implementation of <b>Python Style</b> formatting
248  * is extending the original standard even a little.
249  *
250  *
251  * ##Concatenated Formatters And Concatenated Format Operations##
252  *
253  * In comparison to "usual", similar classes that are found in other libraries/languages, there
254  * are two main differences with this class:
255  * 1. This class supports the concatenation of formatters by passing an instance of
256  * a different derived type in the constructor. A formatter that has a (list of) concatenated
257  * other formatter(s) attached, allows to detect the provided format syntax by passing the format
258  * operation recursively to the concatenated formatter(s). Of-course this approach is limited to
259  * such syntax definitions that are not ambiguous. (...It works well with formatters
260  * \b %FormatterPythonStyle and \b %FormatterJavaStyle!)
261  *
262  * 2. Usually (in other libraries) the interface of formatting methods accept a dedicated
263  * format string and list of arguments to format. Method #Format of this class however
264  * expects just an argument list. The processing then is done as follows:
265  * The first argument in the list is tested to be a format string. If it is, method #format
266  * will report how many arguments form the list have been "consumed" and formatting will continue
267  * with the first argument after the last consumed one. If the actual argument is not identified
268  * as a format string (tested with each concatenated formatter), the string is just appended to
269  * the target and formatting continues with the next argument in the list.<br>
270  * If at the end of this processing loop a single argument is left, then this is not checked
271  * for being a format string. Instead, it will be just appended to the target as is.
272  * An argument that potentially could be a format string is simply skipped (ignored) if it
273  * evaluates to \c nullptr.
274  *
275  * This behaviour has the following consequences:
276  * - No format string needs to be passed. In this case, the arguments are appended to the target
277  * \b %AString in their sequence order
278  * - More than one format string (with according) arguments might be passed.
279  * - The different format strings might even follow a different syntax.
280  *
281  * The biggest \b advantage of this implementation is that format strings and arguments might be
282  * collected prior to invoking the formatter. Collection might happen at different places and times
283  * in a process with the only restriction that not more or less arguments might be added to the
284  * overall list than referred to in the format string. In other words, format strings have to
285  * be well set in respect to the arguments they consume.
286  *
287  * The biggest <b>potential issue</b> of this approach is that a string argument that is not meant
288  * to be a format string might be accidentally interpreted as one if it happens to contain one or
289  * more placeholder characters that are matched by one of the concatenated formatters. (In this case it
290  * it is likely that the format is invalid and an error is written. In debug mode, even an assertion
291  * might be raised). To avoid such ambiguity, the following considerations have to be taken into
292  * account when passing arguments:
293  * - Ambiguities might only happen with string-type arguments. Other types that are converted to
294  * string representations are never considered a potential format string.
295  * - Strings that contain a newline-character (ascii \b 0x0a) are not considered to be a format string.
296  * (Newline characters in a format string have to be escaped using a formatter-specific notation,
297  * e.g. \c "\n").
298  * - A string-type argument that is the last in the list (or the only one) is safe
299  * (no ambiguity can occur) because the last argument is never checked for being a format string.
300  * - %String-type arguments that do not contain placeholder characters defined by one of the formatters
301  * are safe.
302  *
303  * \attention
304  * <b>The rule that results from this is:</b><br>
305  * If none of the conditions mentioned above can be guaranteed, then the argument in question
306  * should be passed together with a preceding simple formatting string, for example,
307  * \c "{}" for \b %FormatterPythonStyle or
308  * \c "%s" in case of \b %FormatterJavaStyle.
309  *
310  * ##Choosing The %Formatter##
311  * As elaborated above, there are two built-in formatter types and those can be concatenated.
312  * Finally, there is the option to implement an own, custom formatter. For the undecided, the
313  * recommendation is to use <b>Java Style</b> for simple, quick <c>printf-</c> like formatting
314  * and to choose <b>Python Style</b> in the moment formatting requirements get more complex.
315  * A special case and good sample for the use of \alib formatting features is found in the
316  * [ALox Logging Library](https://github.com/AlexWorx/ALox-Logging-Library). Here the special
317  * situation is that \alox itself is a library and hence exposes its formatters used to
318  * users. Therefore \alox (by default) uses a <b>Python Style</b> formatter and a concatenated
319  * <b>Java Style</b> version. This way, simple samples will work from scratch for users of all
320  * three languages!
321  *
322  * ##Usage##
323  * The class is abstract, hence only derived types can be instantiated.
324  * Those types are constructed with optionally providing a concatenated instance of another
325  * derived type. The optional concatenated formatter object has a lower priority in respect to
326  * detecting a format string.
327  *
328  * Two overloaded methods \b Format exists:
329  * - The first method accepts a target \b AString and a variadic list of arguments. This method
330  * can be used if the format string(s) and all arguments are available/known at the point of
331  * invocation.
332  * - The second method accepts a target \b AString and a list of arguments. This method
333  * can be used if format string(s) and arguments are collected prior to the final invocation
334  * of the formatter.
335  *
336  * \note
337  * The use of this class is not "thread safe".
338  *
339  * ##Implementing A Custom %Formatter##
340  * To implement a custom formatter, the abstract methods found in this class have to be implemented.
341  * With class
342  * \ref aworx::lib::strings::format::FormatterStdImpl "FormatterStdImpl" a - still abstract -
343  * descendant class is provided. This class implements method #format but introduces a set of
344  * new abstract methods to overwrite. Although, those are more methods, these methods are probably
345  * more simple to implement and therefore deriving from this class might be considered for new
346  * custom types.
347  *
348  * The two built-in <b>Formatter</b>s provided with \alib,
349  * \alib{strings::format,FormatterPythonStyleBase,FormatterPythonStyle} and
350  * \alib{strings::format,FormatterJavaStyleBase,FormatterJavaStyle} derive from
351  * \b %FormatterStdImpl and copying the source code of one of them might be a good start for
352  * implementing a custom type.
353  *
354  * @tparam TChar The character type. Implementations for \c char and \c wchar_t are provided
355  * with type definitions \ref aworx::NFormatter and \ref aworx::WFormatter.
356  **************************************************************************************************/
