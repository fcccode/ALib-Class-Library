22 /** ************************************************************************************************
23  * # Introduction # {#alib_namespace_strings_propertyformatter_overview}
24  *
25  * This class can be used to offer customized format strings to end users.
26  * For this, the format string definition of \alib
27  * \alib{strings::format,FormatterBase,Formatter} types is extended to support identifiers,
28  * which act as placeholders for object values. The identifiers placed in the string have to match
29  * to the normal formatting placeholders in respect to their total number and value type.
30  *
31  * In the constructor of this class, a custom format string which is based on standard \alib
32  * format strings is passed. This string is processed as follows:
33  * - all custom identifiers are removed
34  * - for each identifier found, a reference to an associated callback function is stored.
35  *
36  * Hence, the translation of the custom format string into \alib formatter strings and the
37  * selection of the callback functions is done only once.
38  *
39  * The only interface method #Format then accepts a target string object and a
40  * \ref aworx::lib::boxing "boxed object" of custom type.
41  *
42  * A vector of tuples is defined and needs to be passed to the constructor of this class.
43  * The first element of each tuple contains the \e 'identifier' string which is
44  * removed from the format string if found. The second element contains a callback function used
45  * to retrieve the data when the identifier is found in the format string.
46  *
47  * The whole approach is useful in cases where an application wants to allow a customizable output
48  * format of data objects.
49  *
50  * # Sample # {#alib_namespace_strings_propertyformatter_sample}
51  * Let us assume, an application uses an enumeration and a simple struct:
52  * \snippet "DOX_ALIB_STRINGS.cpp" DOX_ALIB_STRINGS_PROPERTY_FORMATTER_1
53  *
54  * In the application's configuration file, a custom output format for objects of type \b Person
55  * should be user defined. We document to the user that he can use Python-style or
56  * Java-style output formats - extended by the ability to place the following three identifiers in
57  * the string:
58  * - \@name
59  * - \@age
60  * - \@hobby.
61  *
62  * A possible format string then would be:
63  *
64  * "{@name} is aged {@age} and his/her hobby is {@hobby}"
65  *
66  * To tell this class how to retrieve the replacement values, we need to define three callback
67  * functions:
68  * \snippet "DOX_ALIB_STRINGS.cpp" DOX_ALIB_STRINGS_PROPERTY_FORMATTER_2
69  * It is a good idea to place the callback functions in an anonymous (sub-) namespace as they
70  * are only referred to once (in the next step).<br>
71  * As this sample is very trivial, the second parameter <c>AString&</c> is not used and named.
72  * Details on the purpose and use of this parameter is addressed
73  * \ref alib_namespace_strings_propertyformatter_callbacks "below".
74  *
75  * As a next step, these functions need to be collected together in a "translation table".
76  * The table primarily holds a string denoting the replacement identifier and a pointer to the
77  * corresponding callback function. For convenience, the table type is provided with <c>using</c>
78  * defintion #TCallbackTable.
79  *
80  * In our sample, the definition of the table looks like this:
81  * \snippet "DOX_ALIB_STRINGS.cpp" DOX_ALIB_STRINGS_PROPERTY_FORMATTER_3
82  *
83  * This is all we need! Of-course, an external declaration of our table \b PersonCallbacks
84  * should be placed somewhere in the projects' header file. With this, a code like this may now use
85  * the custom formatter strings:
86  * \snippet "DOX_ALIB_STRINGS.cpp" DOX_ALIB_STRINGS_PROPERTY_FORMATTER_4
87  *
88  * Running the code above produces the following output:
89  * \snippet "DOX_ALIB_STRINGS_PROPERTY_FORMATTER.txt" OUTPUT
90  *
91  * Serializing objects in a custom format (e.g. a user could define its own \b JSon object output)
92  * should be the most obvious and frequent use case. And this is what the classes name suggests.
93  * Nevertheless, the callback methods might be more complex than just returning "properties" of
94  * objects. The next sections gives more details on the custom callbacks.
95  *
96  * \~Comment * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * \~
97  * # Callback Functions # {#alib_namespace_strings_propertyformatter_callbacks}
98  * The call back function's signature is defined with <c>using</c> definition #TCallback which
99  * evaluates to
100  *
101  * Box (*)(const Box&, AString&)
102  *
103  * The return type is \ref aworx::lib::boxing::Box "Box", which allows the callback function to
104  * return objects of arbitrary type, respectively all types that received support
105  * with \alib formatters.
106  *
107  * The first input parameter provides the data object passed to method #Format.
108  * If the callback functions are dedicated to a property formatter that receives a certain
109  * object type (what they usually are), the custom type can be straightly unboxed.
110  *
111  * The second parameter is an \b AString object which optionally can be used to assemble
112  * string objects in more complex callback methods. Note, that if this string is not
113  * empty after the callback invocation, the result is copied to a string buffer allocated
114  * in the heap memory. In this case, the box object returned by the callback is ignored
115  * and instead the copy of the string is passed to the formatter. If C++ string constants
116  * (e.g <c>"true"</c> are to be returned by the callbacks, those constants do not need
117  * to be copied to the buffer, as their memory is statically allocated. In other words,
118  * the \b %AString buffer provided in the second parameter is needed to be used
119  * only in cases that a string gets assembled in the callback function!
120  *
121  * \~Comment * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * \~
122  * # Identifiers And The Escape Character '@' # {#alib_namespace_strings_propertyformatter_escape}
123  * The escape character used to find custom identifiers in the format string is defined
124  * with member #ESCCharacter and defaults to \b '\@'.
125  * This escape character is searched in the format string. If found, the identifier is
126  * read by consuming alphabetic characters. If a 'closing' second occurrence of \b '@' is found
127  * while the identifier is parsed, then this character
128  * gets eliminated from the custom format string as well. Hence, the two format strings
129  *
130  * "{@name}"
131  * "{@name@}"
132  * are both valid and equal.<br>
133  *
134  * In case of
135  * \alib{strings::format,FormatterPythonStyleBase,FormatterPythonStyle} format strings,
136  * it is a good idea to place the identifier right inside the brackets. It just looks very intuitive.
137  * However, these versions:
138  *
139  * "@name{}"
140  * "{}@name"
141  * are also valid custom format strings.
142  *
143  * In case of
144  * \alib{strings::format,FormatterJavaStyleBase,FormatterJavaStyle}, which uses \b '\%'
145  * as its escape character, we consider the best option to put the custom identifier in front
146  * of each \b '\%' symbol. The string of the example given above would then look like this:
147  *
148  * "@name%s is aged @age%d and his/her hobby is @hobby%s"
149  * Doing it this way, the \b '\%' symbol acts as a very natural delimiter for the custom identifier.
150  *
151  * As a final note, with the second element of tuple \ref TCallbackTableEntry, it is possible
152  * to allow abbreviations of identifier names. This column denotes the minimum
153  * number of characters to be matched. As in the above sample a value of \c 1 is provided, each
154  * identifier of the custom format string can be abbreviated down to one character. Consequently
155  * the following format string samples are all equal and allowed:
156  *
157  * "{@name}"
158  * "{@nam}"
159  * "{@na}"
160  * "{@n}"
161  * <p>
162 \~Comment ####################################################################################### \~
