289  /** ****************************************************************************************
290  * Finalizes Initialization and has to be called after all invocations of:
291  * - DefineCommands,
292  * - DefineOptions,
293  * - DefineParameters and
294  * - DefineExitCodes,
295  *
296  * have been performed. All options recognized get collected in lists #OptionsFound and
297  * #OptionsOverwritten. The arguments of the options get removed from #ArgsLeft.
298  *
299  * In case of options that have own parameter arguments, such arguments may not be fully
300  * removed. This depends on whether it is possible with the simple flags and values
301  * provided in \alib{cli,OptionDecl} to enable class \alib{cli,Option} to fully detect
302  * them. Therefore, after this method is invoked, for options with more complex syntax,
303  * custom code may be needed to pull the "remainders" of option arguments from #ArgsLeft.
304  * For this, field \alib{cli,Option::Position} is quite useful, as well as method
305  * #RemoveArg.
306  *
307  * It has to be assured that before the next step, which is the invocation of
308  * #ReadNextCommands, all option-related CLI arguments are cleaned away!
309  *
310  * For this reason, this method removes all arguments that start with a
311  * hyphen \c '-' from the #ArgsLeft, even if they got \e not recognized. Those CLI arguments
312  * get collected in #OptionArgsIgnored.
313  * Finding unrecognized options is not considered an error, because other libraries used
314  * with the software might read CLI options autonomously.
315  *
316  * \note
317  * A good sample for this is class \alib{config,CLIArgs} used with \alib
318  * configuration variables. After this method has been invoked, vector #OptionArgsIgnored
319  * may/should be passed to the plug-in \b %CLIArgs of (all)
320  * \alib{config,Configuration} object(s) used with the application. For this, an
321  * invocation, similar to this may be used with all \alib{lang,Library,libraries} that use
322  * an own configuration object:
323  *
324  * XYZLibrary.Config->GetPluginTypeSafe<aworx::lib::config::CLIArgs>()->SetArgs( &OptionArgsIgnored );
325  *
326  * In the case that other libraries have more complex option syntax, e.g. options
327  * consisting of multiple arguments or such that do not even start with a hyphen character,
328  * then, this method should be called <b>only after a custom code removes such 3rd party
329  * options in a reliable way!</b>
330  *
331  * If all this was not done, the "remainder" of custom options might confuse parsing
332  * of commands and its parameters and most probably would lead to an
333  * "unknown" command error when the remainders of non-removed 3rd party option arguments
334  * are consumed later.
335  *
336  * As a consequence of this approach, a subsequent call to this method has no effect.
337  ******************************************************************************************/
338  ALIB_API
339  void ReadOptions();

341  /** ****************************************************************************************
342  * Parses as many commands as possible and stores them in #CommandsParsed. Prior
343  * to invoking this methods, all CLI declarations have to be made. Furthermore, usually
344  * method #ReadOptions has to be invoked prior to this method.
345  *
346  * The name of the method indicates that one or more, but maybe not all commands are read.
347  * The reason for this behaviour lies in the fact that commands may be defined that
348  * do their own, specifically coded parsing. As a matter of the fact that the flags and
349  * options of structs \alib{cli,CommandDecl} and \alib{cli,ParameterDecl} are kept rather
350  * simple to match the most usual cases, the parsing of arguments of a command often
351  * has to be left to be done by custom code. Mostly just when processing (executing) a
352  * command. In contrast to the need of parsing (and processing) all CLI options,
353  * given prior to processing commands, this is not a problem. The usual inner part of a
354  * command processing loop then looks like this:
355  * - Check if #CommandsParsed is empty
356  * - Invoke this method (\b %ReadNextCommands )
357  * - If still no command is found, break loop
358  * - Remove and process first command in #CommandsParsed.
359  *
360  * A similar parsing approach is supported with method #NextCommand. The only difference
361  * is that the parsed commands stay in #CommandsParsed and instead field #NextCommandNo
362  * indicates the position of the next command to read. This way, commands may refer
363  * to previous ones, if this is needed.
364  *
365  * As a final note it should be mentioned, that implementing a "dry run" option on the
366  * level of command parsing, for the reasons explained above, might need some specific
367  * custom code to be able to parse all commands (without processing them). If such
368  * functionality is wanted, it is best to separate custom command parsing from
369  * command execution (the opposite was recommended above). Only the last command in the list
370  * has to be 'manually' processed, as the previous ones obviously got parsed well.
371  * With this approach, all commands can be parsed without executing them. Static utility
372  * method \alib{cli,CLIUtil::DumpParseResults} is a predefined way to then write information
373  * about all options and commands parsed.<br>
374  * A lower level "dry run", that receives information about the concrete actions that the
375  * processing of commands would perform, is of-course a different, completely application
376  * specific task.
377  ******************************************************************************************/
378  ALIB_API
