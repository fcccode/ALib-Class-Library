ALib Symbols To Select Code Fragments

Detailed Description

Symbols (C++ preprocessor macros) listed here, are used internally to select ALib library code and can be used with custom code for the same purpose.

The symbol's values are mostly generated out of ALib Compilation Symbols. In this case, their name usually corresponds to a pair of compilation symbols, excluding the suffixes _ON and _OFF.

Note
ALib volunteers to strictly distinguish between preprocessor symbols that are to be passed to the compiler (usually using CLI-option "-D") and preprocessor symbols that are defined in header files.

ALib Module Selection

This group of symbols determine the availability of different ALib Modules of the ALib distribution used.
The symbols are defined in header alib/distribution.hpp, which varies per distribution.

For module ALib Singleton no symbol is defined, as this module is included in any ALib distribution.

ALox Log-Code Selection

This group of code selection symbols is defined with the use of module ALox and used to select the inclusion of log code. The symbols are seldom to be used by end-user code. Instead, the macros used for the log statements themselves (see ALox Macros For Debug Logging and ALox Macros For Release Logging) are variably defined based on these symbols.

◆ ALIB_FEAT_BOXING_FTYPES

#define ALIB_FEAT_BOXING_FTYPES

Selects code for boxing fundamental types. When using ALib Boxing as a single module, built-in fundamental type boxing may be disabled and replaced by a custom implementation. When using ALib Boxing in combination with ALib Strings, such disabling is not allowed as the formatter classes rely on the default implementation. The same is true for other parts of ALib.

Use ALIB_FEAT_BOXING_FTYPES_OFF to set this symbol to 0 and disable default fundamental types boxing provided by ALib.

If built-in fundamental type boxing is suppressed, then either fundamental types are not boxable or boxing of fundamental types is implemented by the user of the library. The typical reason for providing a custom implementation for boxing fundamental types is that the default implementation provided with ALib, is not injective in a mathematical sense.

See also
Documentation of namespace aworx::lib::boxing::ftypes for more information.

◆ ALIB_FEAT_BOXING_STD_VECTOR

#define ALIB_FEAT_BOXING_STD_VECTOR

Selects code for boxing type std::vector<T, std::allocator<T>> as plain C++ array types "<c>T[]</c>".
Use ALIB_FEAT_BOXING_STD_VECTOR_OFF to set this symbol to 0 and disable this default behaviour. If switched off, the default boxing of custom types applies, which means that values and pointers of type std::vector are boxed as pointers to std::vector object given.

See also
Namespace documentation chapter 4.1 Boxing Arrays and Vectors.

◆ ALIB_FEAT_SINGLETON_MAPPED

#define ALIB_FEAT_SINGLETON_MAPPED

Affects implementation of class Singleton. If this symbol represents true, then a static hash map is used to find the singleton objects, even if different data segments exist within one software process.
While the overhead imposed by such implementation is quite small, with this symbol being false, the overhead is omitted in scenarios when such effort is not necessary.

The symbol defaults to true on the windows platform, otherwise to false. The default value can be changed by passing one of the symbols ALIB_FEAT_SINGLETON_MAPPED_OFF or ALIB_FEAT_SINGLETON_MAPPED_ON to the C++ compiler when building the library.

◆ ALOX_DBG_LOG

#define ALOX_DBG_LOG

If defined as true, debug Log Statements are enabled. Hence, users of ALox can conditionally compile special debug logging code that normally belongs to corresponding debug logging statements. This is useful for example, to conditionally compile code that calculates and gathers information to do some more complex log output.

See ALOX_DBG_LOG_ON and ALOX_DBG_LOG_OFF for information about how to manipulate this symbol.

◆ ALOX_DBG_LOG_CI

#define ALOX_DBG_LOG_CI

If defined, scope information is passed to ALox on invocation of debug Log Statements. Hence, users of ALox can conditionally compile special debug logging code that normally belongs to corresponding debug logging statements based on this symbol. A sample would be the definition of different log line meta information formats, depending on the availability of scope information.

Note
The ALox Scope Domain mechanism as well as the ALox features Lox.Once and Log Data rely on caller information. These mechanisms are therefore not available to debug logging unless this symbol is set.

See ALOX_DBG_LOG_CI_ON and ALOX_DBG_LOG_CI_OFF for information about how to manipulate this symbol.
