aworx::lib::strings::thirdparty::qt Namespace Reference

Programmer's Manual

With the inclusion of header file alib/compatibility/qt.hpp some compatibility between ALib Strings and types of QT Class Library is given.

1. Specializations of T_String:
Specializations of aworx::lib::strings::T_String is given for types

  • QByteArray and
  • QLatin1String This allows to implicitly constructs objects of class String from the named types.

2. Applicable Types:
Support for applying (appending contents of objects of the types to objects of class AString) is given. Applicable types are:

  • QChar
  • QLatin1String
  • QByteArray
  • QVector<uint>
  • QString

3. Conversion to QString:
Several overloaded methods ToQString are defined to convert ALib Strings to instances of class QString.

General Information On ALib And QT String Compatibility:
ALib string classes are designed to support 3rd-party string types in a non-intrusive fashion. Template meta programming in two areas makes this possible:

  • Implicit conversion of external strings to class String, which allows to pass 3rd party strings to method accepting parameters of type const aworx::String& without conversion.
  • Application of arbitrary types to objects of class AString. Here, the term "application" most means concatenating a string representation of an object to the AString object. (This is all documented in reference manual of namespace aworx::lib::strings.)

String classes of the QT library are supported trough header file "alib/compatibility/qt.hpp".

Implicit Conversion:
Unfortunately, objects of class QString can not be implicitly converted (embedded) in thin and lightweight class String. The reason for this is, that ALib strings are single byte strings, while QString represent (two byte) unicode strings. Therefore, template struct T_String can not be implemented for class QString and objects of this class can not be simply passed to methods accepting parameters of type const aworx::String&.

The way out is the use of classes QByteArray and QLatin1String. For these QT classes, an implicit conversion exists and this way objects of these types are implicitly converted to aworx::String types. Class QString provides methods to convert to these classes. Passing a QString object to a method expecting a const aworx::String& can be done as follows 

 MyMethodExpectingALibString( myQString.toLocal8Bit() );
 MyMethodExpectingALibString( myQString.toUtf8()      );
 MyMethodExpectingALibString( myQString.toLatin1()    );
Note
The encoding of strings represented by ALib strings is user defined. Therefore, it is up to the user to decide which of the above conversion methods of QString is to be used. In most cases, method toLocal8Bit should be the right choice. Also, ALib strings uses the current locale to convert characters of type wchar_t (or arrays of them) when applying to class AString as well.

The obvious drawback of this approach is that a conversion has to take place and that this conversion implies a temporary allocation of heap memory for the data stored in the (stack-allocated) QByteArray object.
