55 /** ************************************************************************************************
56  * This class manages an object of internal type
57  * \ref aworx::lib::util::StringTree::Node "Node", which consists of
58  * - A value of custom (template) type \p{T},
59  * - A pointer to a parent \b %Node and
60  * - A hash table which recursively stores pointers to \b Node objects, referred to by
61  * a string-type key.
62  *
63  * This recursive data structure is what comprises a graph with named edges. As this object does
64  * not allow insertion of nodes with circular dependencies, the graph is of tree type.
65  * The way from the root node to the leafs usually is called "path" and the class incorporates
66  * functionality to work with string representations of such paths where names of edges are
67  * concatenated and separated by a special separation character.
68  * Now, this explains why this class is called \b %StringTree.
69  *
70  * The search and creation of tree nodes by using aforementioned path strings, is very similar to
71  * what is well known from addressing files and folders in file systems.
72  * The difference is however, that this class does not differentiate between 'folders' and 'files',
73  * hence between 'nodes' and 'leafs'. Every node has the same value type \p{T} and may or may not
74  * have child nodes.
75  * If such differentiation - or other semantics - is wanted, this may be modeled in custom
76  * attributes provided in template type \p{T}.
77  *
78  * The internal node structure is not exposed publically and the class has no direct interface to
79  * manipulate nodes. Instead, the interface is defined by three public inner types.
80  * Those are:
81  * - \ref aworx::lib::util::StringTree::Cursor "Cursor",
82  * - \ref aworx::lib::util::StringTree::StdIterator "StdIterator" (C++ version only) and
83  * - \ref aworx::lib::util::StringTree::Walker "Walker".
84  *
85  * and are explained in the following paragraphs.
86  *
87  * \anchor cpp_alib_containers_stringtree_cursor
88  * <b>Inner class %Cursor: Inserting, Retrieving And Deleting Nodes</b><br>
89  * The class has no direct interface to manipulate nodes. The main interface into
90  * objects of this type is defined by public, inner type
91  * \ref aworx::lib::util::StringTree::Cursor "Cursor". The only way to create an
92  * initial cursor is with method #Root, which creates a cursor object referring to the
93  * root node of the tree. With this, string names and complete paths can be used to
94  * move the cursor along existing nodes of the tree or to create new child nodes or a whole
95  * path of children at once.
96  * Class cursor is quite lightweight as it contains just two pointers (to the \b %StringTree and the current node).
97  * Hence, cursors can be cloned, assigned and passed around very efficiently.
98  *
99  * Once a cursor is created it can traverse over the tree nodes using methods
100  * - \ref aworx::lib::util::StringTree::Cursor::MoveToChild "Cursor::MoveToChild",
101  * - \ref aworx::lib::util::StringTree::Cursor::MoveToParent "Cursor::MoveToParent",
102  * - \ref aworx::lib::util::StringTree::Cursor::MoveTo "Cursor::MoveTo" and
103  * - \ref aworx::lib::util::StringTree::Cursor::MoveToExistingPart "Cursor::MoveToExistingPart".
104  *
105  * For the creation of new child nodes or a complete path of such, methods
106  * - \ref aworx::lib::util::StringTree::Cursor::MoveToChildCreate "Cursor::MoveToChildCreate" and
107  * - \ref aworx::lib::util::StringTree::Cursor::MoveToAndCreateNonExistingPart "Cursor::MoveToAndCreateNonExistingPart"
108  *
109  * are provided. Then, two methods for erasing nodes exist:
110  * - \ref aworx::lib::util::StringTree::Cursor::DeleteChild "Cursor::DeleteChild" and
111  * - \ref aworx::lib::util::StringTree::Cursor::DeleteChildren "Cursor::DeleteChildren"
112  *
113  * Finally, class cursor can also be used like an iterator. But it explicitly and for good reason
114  * does \b not follow the concepts provided with <c>C++ std::iterator</c> types. The iterating
115  * feature comes with methods
116  * - \ref aworx::lib::util::StringTree::Cursor::MoveToParentUnchecked "Cursor::MoveToParentUnchecked" and
117  * - \ref aworx::lib::util::StringTree::Cursor::IsValid "Cursor::IsValid".
118  *
119  * The first transitions the cursor to refer to the parent node. Using this method is (almost)
120  * the only way how a cursor can get invalidated. Hence a simple loop can be written that visits
121  * all nodes from a starting node, up to the root of the tree until \b %Cursor::IsValid evaluates
122  * to \c false.
123  *
124  * <b>Inner class iterator: A std::iterator implementation</b><br>
125  * A second inner class is provided with
126  * \ref aworx::lib::util::StringTree::StdIterator "StdIterator". It implements the concept
127  * [C++ InputIterator](http://en.cppreference.com/w/cpp/concept/InputIterator) and due to the
128  * implementation of methods
129  * - \ref aworx::lib::util::StringTree::begin "StringTree::begin" and
130  * - \ref aworx::lib::util::StringTree::end "StringTree::end",
131  *
132  * this type can be used together with <em>C++ 11 ranged-for
133  * loops</em> as well as \c std::for_each. And these loops indicate already the typical use of
134  * class \b %StdIterator: The implementation of algorithms that need to visit all nodes of a tree or a
135  * branch in an depth-first order, with unsorted child order.
136  *
137  * With methods
138  * - \ref aworx::lib::util::StringTree::Cursor::begin "Cursor::begin" and
139  * - \ref aworx::lib::util::StringTree::Cursor::end "Cursor::end",
140  *
141  * StdIterator objects may be initialized using a different start node than the root, which
142  * is chosen with the direct \b %StringTree interface.
143  * Existing \b %StdIterator object can also be reused: They can be freshly initialized with methods
144  *
145  * - \ref aworx::lib::util::StringTree::Cursor::SetIterator "Cursor::SetIterator" and
146  * - \ref aworx::lib::util::StringTree::Walker::SetIterator "Walker::SetIterator".
147  *
148  * Please also note the paragraph about recursive iteration below.
149  *
150  * Class \ref aworx::lib::util::StringTree::StdIterator "StdIterator" is quite lightweight.
151  * Nevertheless, if frequently used, it might be advisable
152  * to reuse an exiting \b %StdIterator, because internal buffers are allocated then only once. Of-course,
153  * re-using an instance is a showstopper for convenient <em>C++ 11 ranged-for loops</em>.
154  *
155  * <b>Inner class %Walker: Sorted Iterations</b><br>
156  * Finally, with
157  * \ref aworx::lib::util::StringTree::Walker "Walker", a third \b %StdIterator class is exposed.
158  * The rational for having this class is to provide a configurable and controlled way of iterating
159  * a tree or a branch. Some features of the class are:
160  * - Can be created from (or reset to) the \b %StringTree itself,
161  * \ref aworx::lib::util::StringTree::Cursor "Cursor" objects and from
162  * \ref aworx::lib::util::StringTree::StdIterator "StdIterator" objects.
163  * - Can work recursively (with depth limitation) or just on the current nodes' direct children.
164  * - Has several sort options:
165  * - built-in by path name, ascending/descending, case sensitive or ignoring
166  * - user defined by path name
167  * - user defined by template type
168  *
169  * The class is rather 'heavy' and recursive iteration needs to allocate memory for a sorted vector
170  * of child nodes.
171  * Therefore it is recommended to reuse instances of the class with subsequent, similar iterations.
172  * In addition this explains why this class does not follow the concept of <c>std::iterator</c>,
173  * same as class
174  * \ref aworx::lib::util::StringTree::Cursor "Cursor" does not - but then unlike class
175  * \ref aworx::lib::util::StringTree::StdIterator "StdIterator" which does!
176  *
177  * \anchor cpp_alib_containers_stringtree_iterator
178  * <b>Recursive Iteration</b><br>
179  *
180  * Both inner types that provide depth-iteration, namely class
181  * \ref aworx::lib::util::StringTree::StdIterator "StdIterator" and class
182  * \ref aworx::lib::util::StringTree::Walker "Walker", allow recursive iteration. There are
183  * two specialities to note on this topic which need explanation:
184  *
185  * 1. <b>Skipping the Initial %Node:</b><br>
186  * When the recursive iteration is limited, then the very first node (the root node of the
187  * iteration) is skipped. With unlimited recursion, the root node is included (as the first
188  * result). In the case, that the root node should not be included in an unlimited, recursive
189  * iteration, it has to be skipped 'manually'.
190  * 2. <b>Default Recursion Depth:</b><br>
191  * There are different default values for the recursion limit, depending on the way a \b %StdIterator
192  * is initialized.
193  * For example, method \b %StringTree::begin() sets up an recursive \b %StdIterator, while
194  * \b %Cursor::begin limits the recursion to \c 1, hence to iterate only the direct child
195  * nodes. This default behavior was identified to be reasonable for many situations.<br>
196  * If different recursiveness is wanted, a parameter needs to be passed to \b %begin, which of
197  * course is then not applicable to nice and short <em>C++ 11 ranged-for loops</em>.
198  *
199  * <b>Iterators And Changes Of The Underlying Container</b><br>
200  * As with container classes in namespace \c std, objects of iterator-like types of this class (i.e.
201  * \ref aworx::lib::util::StringTree::Cursor "Cursor",
202  * \ref aworx::lib::util::StringTree::StdIterator "StdIterator" and
203  * \ref aworx::lib::util::StringTree::Walker "Walker") may become invalid when
204  * the underlying \b %StringTree object changes and have to be re-initialized after such change.
205  * I.e, there is no guarantee given that changes in the tree are not invalidating iterators that
206  * represent 'higher' nodes or or 'sibling' branches. Such invalid state can not be detected. As
207  * a result, all iterator like objects have to be reset or disposed on changes of the tree.
208  *
209  * TOODX: reflect changes. Better to re-read everything here, also the three allocator classes!
210  *
211  * <b>Key String Allocations:</b><br>
212  * The C++ version of this class implements user-defined allocation of the characters sequences
213  * stored as keys in the child hash tables of each node. For this, template parameter
214  * \p{StringTreeDefaultAllocator} is given which defaults to struct
215  * \ref aworx::lib::util::StringTreeDefaultAllocator "StringTreeDefaultAllocator", which
216  * just uses new and delete to allocate and free the space.
217  * This is done, because chances are good that
218  * - either the key values provided in the child creation methods of class \b %Cursor are
219  * permanently valid or
220  * - the templated value element \p{T} contains a "copy" of the key.
221  *
222  * If the first case applies, type
223  * \ref aworx::lib::util::StringTreeConstKeyAllocator "StringTreeConstKeyAllocator" may be used,
224  * which does not allocate anything and does not copy the key even.<br>
225  * If the second case applies, a custom type may be provided which stores the key in \p{T} and
226  * returns this copy to be used. Note that for this reason, class \b StringTree asserts that
227  * node elements and with them custom objects of type \p{T} are deleted always \b after they have
228  * been removed from the tree, so that the key memory can safely freed.
229  *
230  * @tparam T The value type of elements stored in the nodes of the tree.
231  * @tparam TAllocator Type with allocation methods as defined in default struct
232  * \alib{util,StringTreeDefaultAllocator}.
233  * @tparam TSeparator The separation character used with cursor paths.
234  **************************************************************************************************/

499

503  delete[] childNames;
504  }
505  }

804  /**
805  * Finds a child node along the \p{path} given, but does not create new nodes.
806  * Incomplete results may occur if a child along the path was not found.
807  * In this case, parameter \p{path} contains
808  * the remaining path, excluding a leading separator.
809  *
810  * A leading slash (aka \p{TSeparator}) allows absolute path addressing, which means
811  * the root of \p{node} is searched if a leading separator is found.
812  *
813  * Besides normal child names, this method accepts
814  * - multiple separator characters (ignored)
815  * - child name "." (ignored)
816  * - child name ".." for parent node
817  *
818  * If, while processing the path string, the root node is found an the next
819  * path element is "..", this element is ignored and processing continues.
820  * As a sample, the paths:
821  *
822  * /a/../b
823  * and
824  *
825  * /a/../../b
826  * both evaluate to
827  *
828  * /b
829  * assumed that \e /a and \e /b exist.
830  *
831  * @param node The start node.
832  * @param[in,out] path Creation path. Provided as reference and consumed as far
833  * as the path exits.
834  * @return The node found
835  */
836  static
837  Node* followPath( Node* node, Substring& path )
838  {
839  // check for "root addressing"
840  if( path.CharAtStart() == TSeparator )
841  {
842  path.ConsumeChars( 1 );
843  while( node->parent != nullptr )
844  node= node->parent;
845  }
846 
847  // loop over node names in path
848  for(;;)
849  {
850  // multiple separators are ignored
851  while(path.ConsumeChar( TSeparator ) )
852  ;
853 
854  if( path.IsEmpty() )
855  return node;
856 
857 
858  String name= path.Substring<false>( 0, path.IndexOfOrLength( TSeparator ) );
859 
860 
861  if( name.Equals( ASTR("..") ) )
862  {
863  // we move up only if possible. If not, the ".." is just ignored (consumed)
864  // (This behavior was just more helpful in the use cases so far)
865  if( node->parent != nullptr )
866  node= node->parent;
867  }
868 
869  else if( !name.Equals( ASTR(".") ) )
870  {
871  auto it= node->children.find( name );
872  if( it == node->children.end() )
873  return node;
874  node= it->second;
875  }
876 
877  path.ConsumeChars( name.Length() );
878  }
879  }
880 
881  /**
882  * Creates child nodes corresponding to the given \p{path}.
883  * If the first node already exists, nothing is done and \c nullptr is returned as
884  * this is considered an error.
885  * If the given path is empty the given node is returned as this is \b not
886  * considered an error.
887  *
888  * Child names <c>"."</c> and <c>".."</c> are ignored, but considered an error.
889  * In debug builds, an \ref ALIB_WARNING is reported.
890  *
891  * The final leaf node is returned.
892  *
893  * @param allocator The allocator used to create the name key.
894  * @param node The start node.
895  * @param[in,out] path Creation path. Provided as reference and consumed if not
896  * errorneous.
897  * @return The leaf node of all created nodes. \c nullptr in the case that the
898  * first child given in \p{path} already exists.
899  */
900  static
901  Node* createPathAndReturnLeaf( TAllocator& allocator, Node* node, Substring& path )
902  {
903  bool isFirst= true;
904  for(;;)
905  {
906  // consume and check
907  while(path.ConsumeChar( TSeparator ) )
908  ;
909  if( path.IsEmpty() )
910  return node;
911 
912  // get next name
913  String childName= path.Substring<false>( 0, path.IndexOfOrLength( TSeparator ) );
914  if( childName.Equals( ASTR(".") ) || childName.Equals(ASTR("..")) )
915  {
916  //ALIB_WARNING( "Illegal child name {!Q} in creation path", childName );
917  return node;
918  }
919 
920  Node* child;
921  if( isFirst )
922  {
923  child= node->createChild( allocator, childName );
924  if( child == nullptr )
925  return nullptr;
926  }
927  else
928  {
929  child= node->createChildNoCheck( allocator, childName );
930  }
931  isFirst= false;
932 
933  // done?
934  path.ConsumeChars( childName.Length() + 1);
935  node= child;
936  }
937  }

1002  /**

1004  * two different versions of the method can be selected.<br>
1005  * The default, checked version (\p{TCheck} = \c true ) will not move a cursor
1006  * that is already pointing to the root of the \b %StringTree into an invalid state.
1007  * Instead the cursor remains unchanged and \c false is returned.<br>
1008  * The unchecked version will turn the cursor into an invalid state if no
1009  * parent node exists. This is useful in two situations:
1010  * 1. For effective iterating from any node up to
1011  * the \b %StringTree root node. The condition to check in such loop is #IsValid.
1012  * 2. In situations when it is was already checked that a cursor does not represent
1013  * the root node. This is demonstrated in the following sample taken from this
1014  * classes' method #SearchPath:
1015  * \snippet "alib/util/stringtree.hpp" DOX_ALIB_STRING_TREE_MOVETOPARENT

1017  * insertion methods of class \alib{strings,AStringBase,AString} because
1018  * the code prior to the code snippet ensured the validity of all parameters.

1021  * \note If this class is nested in another template class, the invocation of this
1022  * method with the template parameter may be ambiguous. Therefore,
1023  * an alias to the non-checking version is provided with
1024  * #MoveToParentUnchecked.

1027  * If \c <false> is added to the method name, no check is

1031  * otherwise.

1033  * this cursor is valid after the operation.
