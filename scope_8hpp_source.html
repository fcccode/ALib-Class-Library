24 /** ************************************************************************************************
25  * This class acts as a simple virtual container to store custom resources in
26  * \alib{expressions,Scope} objects.
27  *
28  * To do so, custom derived types would simply add a custom member object together with a virtual
29  * destructor that assures that the member object(s) get rightfully deleted.
30  *
31  * Instances of this type allocated with operator \c new, may be stored in class \b %Scope in
32  * containers accessible with fields \alib{expressions::Scope,Resources} and
33  * \alib{expressions::Scope,NamedResources}.
34  * All objects in these fields will be deleted with method \alib{expressions,Scope::Clear}.
35  **************************************************************************************************/

38  virtual ~ScopeResource()
39  {}
40 };

42 /** ************************************************************************************************
43  * This type is used as base class to provide access to program data when evaluating
44  * \alib expressions.
45  * Usually a derived type which contains references to necessary application data is passed to
46  * method \alib{expressions,Expression::Evaluate}.
47  * Then, custom callback functions may cast this object back to its original type and access such
48  * application data.
49  *
50  * Also, scope objects are used to store intermediate results and of-course the final one,
51  * in the case that such results are not of a simple type that can be boxed
52  * \alib{boxing,Box} "by value".
53  *
54  * For this, different simple allocator objects are provided. A custom, derived type may add own
55  * storage facilities, for example vectors of custom objects which are deleted when the vector is
56  * deleted.
57  *
58  * A scope object can be reused for evaluating the same expression several times. Prior to
59  * evaluation, the custom "scoped data" has to be set.
60  * With each reuse, method #Clear will be invoked internally.
61  * Hence, if custom 'allocators' are added in derived types, this method has
62  * to be overwritten to a) invoke the original method and b) clean such custom types.
63  *
64  * One singleton of this type, used to store compile-time data is created with virtual
65  * method \alib{expressions,Compiler::getCompileTimeScope}. If compile-time invokable custom
66  * callback methods use custom storage allocators, this method has to be overridden to return
67  * the proper custom version of this class. (Note, this is not needed for the evaluation-time
68  * instances, as this is created in the custom code unit anyhow and passed to method
69  * \alib{expressions,Expression::Evaluate}.
70  *
71  * \see
72  * For details on the use of scopes see manual chapter
73  * \ref alib_expressions_scopes "8 Scopes".
74  *
75  **************************************************************************************************/
76 struct Scope
77 {
78  /**
79  * This is a pointer to the compile-time scope. This is available only at evaluation time
80  * and primarily is used to access field #NamedResources. This allows to create resources
81  * at compile time, which can be used for evaluation.<br>
82  *
83  * A sample use case is implemented with built-in compiler plug-in
84  * \alib{expressions::plugins,Strings}. When wildcard or regex matching is performed on
85  * constant pattern strings, the matching class (which itself "compiles" the pattern once) is
86  * created once and reused during evaluation.
87  */
88  Scope* CTScope;
